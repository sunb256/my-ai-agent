はい、その構成理解で合っています。おすすめは **agent-core は `yield` で `AgentEvent` を出す、FastAPI はそれを SSE / stream API に変換する、React はイベント種別ごとに UI state を更新する** です。最終回答は「特別扱い」というより、`answer_delta` だけを assistant メッセージ本文として蓄積し、`status` や `tool_call` は進捗・ツール表示として別表示にする、という扱いが自然です。

assistant-ui とも結合できます。ただし、あなたの独自 `AgentEvent` をそのまま assistant-ui が理解するわけではないので、**assistant-ui の Runtime / message parts / tool UI に変換する薄いアダプタ層**を作るイメージです。assistant-ui には custom backend 向けに `LocalRuntime`、独自 state 管理向けに `ExternalStoreRuntime`、標準化された stream 形式向けに Data Stream Protocol が用意されています。`LocalRuntime` は custom backend 接続の最短経路として `run` または `async *run` を実装する形、`ExternalStoreRuntime` は自分で message 配列と callback を持つ形です。Data Stream Protocol は text delta、tool call、error handling、cancellation などを含む message-streaming protocol と説明されています。([assistant-ui][1])

まず agent 側はこういうイベント型にしておくと、例外処理も扱いやすいです。

```python
from dataclasses import dataclass, field
from typing import Any, AsyncIterator, Literal


AgentEventType = Literal[
    "status",
    "tool_call_started",
    "tool_call_finished",
    "answer_delta",
    "answer_done",
    "error",
    "done",
]


@dataclass(frozen=True)
class AgentEvent:
    type: AgentEventType
    message: str | None = None
    data: dict[str, Any] = field(default_factory=dict)
```

例外処理を含めた `yield` 実装は、基本的には `try/except/finally` で書けます。

```python
class Agent:
    async def run_agent_stream(
        self,
        ctx: ExecContext,
        req: Request,
    ) -> AsyncIterator[AgentEvent]:
        try:
            yield AgentEvent(
                type="status",
                message="モデルに問い合わせています",
            )

            while ctx.current_step < self.max_steps:
                response = await self.llm.call_llm(req)

                if response.err_msg:
                    yield AgentEvent(
                        type="error",
                        message=response.err_msg,
                    )
                    return

                if response.tool_calls:
                    for tool_call in response.tool_calls:
                        yield AgentEvent(
                            type="tool_call_started",
                            message=f"{tool_call.name} を実行しています",
                            data={
                                "tool_call_id": tool_call.tool_call_id,
                                "name": tool_call.name,
                                "args": self._safe_tool_args(tool_call),
                            },
                        )

                        try:
                            tool_result = await self.tools.execute(ctx, tool_call)

                        except Exception as error:
                            yield AgentEvent(
                                type="error",
                                message=f"{tool_call.name} の実行に失敗しました",
                                data={
                                    "tool_call_id": tool_call.tool_call_id,
                                    "name": tool_call.name,
                                    "error": str(error),
                                },
                            )
                            return

                        yield AgentEvent(
                            type="tool_call_finished",
                            message=f"{tool_call.name} が完了しました",
                            data={
                                "tool_call_id": tool_call.tool_call_id,
                                "name": tool_call.name,
                                "result_summary": self._summarize_tool_result(tool_result),
                            },
                        )

                        req.contents.append(tool_result)

                    ctx.increment()
                    continue

                if response.text:
                    # まずは一括でもよい
                    yield AgentEvent(
                        type="answer_delta",
                        message=response.text,
                    )

                yield AgentEvent(type="answer_done")
                yield AgentEvent(type="done")
                return

            yield AgentEvent(
                type="error",
                message="最大ステップ数に達しました",
            )

        except Exception as error:
            # 想定外の例外もUIへ流す
            yield AgentEvent(
                type="error",
                message="エージェント実行中に予期しないエラーが発生しました",
                data={"error": str(error)},
            )

        finally:
            # sandbox cleanup / lock release / temp file cleanup など
            await self._cleanup(ctx)
```

ポイントは、異常系を全部 Python 例外として外へ投げるのではなく、**UIに見せたい異常は `AgentEvent(type="error")` として返す**ことです。そうするとReact側は通常イベントと同じループで処理できます。逆に、プログラムのバグやキャンセルはログに残しつつ `error` に変換する、という方針にできます。

FastAPI 側は、`AgentEvent` を JSON Lines や SSE に変換します。ブラウザで扱いやすいのは SSE です。

React 側は、`answer_delta` だけを回答本文に足し、それ以外を進捗欄に入れるのが基本です。

```tsx
type AgentEvent = {
  type:
    | "status"
    | "tool_call_started"
    | "tool_call_finished"
    | "answer_delta"
    | "answer_done"
    | "error"
    | "done";
  message?: string;
  data?: Record<string, unknown>;
};

function applyAgentEvent(event: AgentEvent) {
  if (event.type === "answer_delta") {
    setAssistantAnswer((prev) => prev + (event.message ?? ""));
    return;
  }

  if (event.type === "tool_call_started" || event.type === "tool_call_finished") {
    setToolEvents((prev) => [...prev, event]);
    return;
  }

  if (event.type === "status") {
    setStatus(event.message ?? "");
    return;
  }

  if (event.type === "error") {
    setError(event.message ?? "エラーが発生しました");
    return;
  }
}
```

つまり最終回答は「通常と違う扱いにする」というより、**同じ stream の中に流れてくるが、React側では assistant message の本文として扱う**ということです。

```text
status / tool_call_started / tool_call_finished
  → 進捗・ツール表示

answer_delta
  → assistant の本文

error
  → エラー表示

done
  → streaming 終了
```

assistant-ui と結合する場合は、大きく2パターンあります。

1つ目は、**assistant-ui の外側で独自 stream を処理して、ExternalStoreRuntime に message state を渡す方式**です。あなたが React 側で `messages` を管理し、SSEイベントを受け取るたびに assistant message の parts を更新します。assistant-ui の `ExternalStoreRuntime` は既存の state 管理と接続するための仕組みで、Redux/Zustandなどの外部 store と bridge できると説明されています。([assistant-ui][2])

イメージはこうです。

```tsx
// 疑似コード
const [messages, setMessages] = useState<ChatMessage[]>([]);

async function sendMessage(text: string) {
  addUserMessage(text);
  const assistantMessageId = addEmptyAssistantMessage();

  await readAgentStream(text, (event) => {
    if (event.type === "answer_delta") {
      appendTextToAssistantMessage(assistantMessageId, event.message ?? "");
    }

    if (event.type === "tool_call_started") {
      addToolPart(assistantMessageId, {
        toolName: event.data?.name,
        args: event.data?.args,
        status: "running",
      });
    }

    if (event.type === "tool_call_finished") {
      updateToolPart(assistantMessageId, {
        toolCallId: event.data?.tool_call_id,
        status: "complete",
        result: event.data?.result_summary,
      });
    }

    if (event.type === "error") {
      markAssistantMessageError(assistantMessageId, event.message);
    }
  });
}
```

この方式は、あなたの独自エージェント構造を保ちやすいです。欠点は、assistant-ui が期待する message / part 形式へ変換するコードを自分で書く必要があることです。

2つ目は、**assistant-ui の LocalRuntime / Data Stream Protocol に寄せる方式**です。assistant-ui の `LocalRuntime` は custom backend 接続の簡単な方法として `ChatModelAdapter` の `run` または `async *run` を実装する形です。Data Stream Protocol は assistant-ui が消費できる標準的な streaming format で、text、tool calls、error、cancellation を扱う用途に向いています。([assistant-ui][1])

この場合は、Python/FastAPI 側の stream を assistant-ui が期待する stream 形式へ近づけます。長期的にはこちらの方が assistant-ui の機能に乗りやすいです。

tool表示についても assistant-ui 側に機能があります。assistant-ui の Tool Calling guide は tool interaction の real-time visualization を扱うものとして説明されており、`makeAssistantToolUI` は `toolName` ごとに `args`、`result`、`status` を受け取ってツールUIを描画する API です。また `ToolGroup` は連続する tool call を折りたたみ表示し、streaming 中は自動展開する用途のコンポーネントです。([assistant-ui][3])

したがって、あなたの設計では次の責務分担がきれいです。

```text
agent-core
  run_agent_stream()
  AgentEvent を yield する
  例外を error event に変換する
  tool_call_started / tool_call_finished / answer_delta を出す

agent-api / FastAPI
  AgentEvent を SSE または Data Stream に変換する
  HTTP切断時のキャンセルを扱う
  サーバ側ログを残す

frontend / React
  stream を読む
  answer_delta は assistant message に追加
  tool event は tool UI / progress UI に追加
  error は message error にする

assistant-ui
  message描画
  tool call表示
  streaming中の表示
  thread UI / input UI
```

個人的には、最初から assistant-ui の内部形式に完全準拠しようとしすぎるより、まず agent-core では独自の `AgentEvent` を安定させる方がよいです。その後で、FastAPIまたはReact側に **adapter** を1枚置いて assistant-ui に合わせるのが安全です。

```text
AgentEvent
  ↓ adapter
assistant-ui message parts
```

この分離にしておけば、CLIでは同じ `AgentEvent` をそのまま表示できます。

```python
async for event in agent.run_agent_stream(ctx, req):
    if event.type == "status":
        print(event.message)
    elif event.type == "tool_call_started":
        print(f"[tool] {event.message}")
    elif event.type == "answer_delta":
        print(event.message, end="")
```

Webでは同じイベントをSSEにできます。

```python
async for event in agent.run_agent_stream(ctx, req):
    yield to_sse(event)
```

assistant-uiでは同じイベントを message parts に変換できます。

```tsx
if (event.type === "answer_delta") {
  appendAssistantText(event.message ?? "");
}

if (event.type === "tool_call_started") {
  appendToolCallPart({
    toolName: event.data?.name,
    args: event.data?.args,
    status: "running",
  });
}

if (event.type === "tool_call_finished") {
  completeToolCallPart({
    toolCallId: event.data?.tool_call_id,
    result: event.data?.result_summary,
  });
}
```

例外処理についての不安はかなり妥当ですが、`yield` だから難しいというより、**イベント設計を先に決めると楽になる**タイプです。最低限、`error` と `done` を分けておくとよいです。

```text
error:
  失敗したことをUIへ伝える
  復旧可能/不可能は data に入れる

done:
  streamが正常に終わったことを伝える

answer_done:
  assistant本文の生成が終わったことを伝える
```

`error` のあとに必ず `done` を出すかは設計次第ですが、UI実装を簡単にするなら、**stream終了時は成功・失敗に関係なく最後に `done` を出す**方が扱いやすいです。

```python
try:
    ...
except Exception as error:
    yield AgentEvent(type="error", message=str(error))
finally:
    yield AgentEvent(type="done")
```

ただし `finally` で `yield` すると、キャンセル時の扱いが少し複雑になることがあります。最初は `except` の中で `error` を返し、正常終了時に `done` を返すくらいで十分です。

結論として、あなたの理解している構成でよいです。

```text
agent側:
  yieldで AgentEvent を順番に出す

FastAPI側:
  AgentEvent を SSE / stream API に変換する

React側:
  fetch で stream を読み、event.type に応じて state 更新する

assistant-ui側:
  独自eventを assistant-ui の message / tool parts に変換して表示する
```

最終回答は別APIで取りに行く必要はありません。同じ stream の中に `answer_delta` として流し、React側で assistant message に連結していくのが自然です。進捗やツール表示は同じ stream 内の別イベントとして扱えば、ChatGPTのような「作業中表示」と「最終回答表示」を分けられます。

[2]: https://www.assistant-ui.com/docs/runtimes/custom/external-store?utm_source=chatgpt.com "ExternalStoreRuntime"
[3]: https://www.assistant-ui.com/docs/guides/tools?utm_source=chatgpt.com "Tool Calling — assistant-ui (React Chat UI for AI)"


結論として、React初級〜中級で assistant-ui 初めてなら、最初は **`LocalRuntime + 自作FastAPI stream`** で理解し、その後 **`Data Stream Protocol` に寄せる** のが一番安全です。`Data Stream Protocol` はフロントエンド側がかなり薄くなりますが、そのぶん FastAPI 側が assistant-ui の期待する stream 形式に合わせる必要があります。assistant-ui の公式ドキュメントでも、`LocalRuntime` は custom backend に接続する一番シンプルな方法で、`run` という1つの関数を実装すれば、messages、threads、branching、editing、regeneration、cancellation などは runtime 側が扱う、と説明されています。([assistant-ui][1])

まず全体像はこうです。

```text id="fxwleq"
React画面
  └─ <Thread />                         ← チャットUI本体

assistant-ui Runtime
  └─ useLocalRuntime(MyModelAdapter)     ← UIとAPIをつなぐ層

MyModelAdapter
  └─ fetch("/api/chat/stream")           ← FastAPIを呼ぶ

FastAPI
  └─ agent.run_agent_stream()            ← Python側のyield実装

agent-core
  └─ AgentEventをyieldする
```

assistant-ui を使うときに一番大事なのは、**自分でチャット画面の state を全部持たなくてよい**という点です。assistant-ui は大きく「Frontend components」「Runtime」「Cloud」の3層で構成され、Runtime は UI と backend / LLM をつなぐ状態管理層です。([assistant-ui][2]) つまり、React側で `messages`、入力欄、送信中、キャンセル、再生成などを全部自作するのではなく、`Thread` と `Runtime` に任せます。

## 2. まずは LocalRuntime で作るのが理解しやすい

最初の実装は、Data Stream Protocol に完全準拠させる前に、`LocalRuntime` で自作APIを読む形がわかりやすいです。`LocalRuntime` では `ChatModelAdapter` の `run` を実装します。通常レスポンスなら `async run`、ストリーミングなら `async *run` にします。assistant-ui のドキュメントでも、streaming responses では `async *run` を使い、各 `yield` では差分ではなく累積済みの全文を返す、とされています。([assistant-ui][1])

たとえば、React側にこういう provider を作ります。

```tsx id="odm3e2"
// src/runtime/MyRuntimeProvider.tsx

import type { ReactNode } from "react";
import {
  AssistantRuntimeProvider,
  useLocalRuntime,
  type ChatModelAdapter,

type AgentEvent =
  | {
      type: "status";
      message?: string;
      data?: Record<string, unknown>;
    }
  | {
      type: "tool_call_started";
      message?: string;
      data: {
        tool_call_id: string;
        name: string;
        args?: Record<string, unknown>;
      };
    }
  | {
      type: "tool_call_finished";
      message?: string;
      data: {
        tool_call_id: string;
        name: string;
        result_summary?: unknown;
      };
    }
  | {
      type: "answer_delta";
      message?: string;
      data?: Record<string, unknown>;
    }
  | {
      type: "error";
      message?: string;
      data?: Record<string, unknown>;

      type: "done";
      message?: string;
      data?: Record<string, unknown>;
    };
```

`type AgentEvent = ...` は TypeScript の型定義です。Pythonでいうと、`AgentEvent` の構造を先に宣言しているようなものです。React初心者でもここは「APIから返ってくるJSONの種類を定義している」と考えれば十分です。

次に、FastAPI の SSE を読む関数を作ります。

```tsx id="smic3y"
async function* readSseEvents(
  body: ReadableStream<Uint8Array>,
): AsyncGenerator<AgentEvent> {
  const reader = body.getReader();
  const decoder = new TextDecoder();

  let buffer = "";

  while (true) {
    const { value, done } = await reader.read();

    if (done) {
      break;
    }

    buffer += decoder.decode(value, { stream: true });

    const chunks = buffer.split("\n\n");
    buffer = chunks.pop() ?? "";

    for (const chunk of chunks) {
      const line = chunk
        .split("\n")
        .find((line) => line.startsWith("data: "));

      if (!line) {
        continue;
      }

      const jsonText = line.slice("data: ".length);
      yield JSON.parse(jsonText) as AgentEvent;
    }
  }
}
```

これは React というより、ブラウザの `fetch stream` を読む処理です。Python側の `yield AgentEvent(...)` が FastAPI で SSE になり、React側ではそれを `for await` で読む、という対応になります。

次に `ChatModelAdapter` を書きます。

```tsx id="rk4sdk"
const MyModelAdapter: ChatModelAdapter = {
  async *run({ messages, abortSignal }) {
    const response = await fetch("/api/chat/stream", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify({
        messages,
      }),
      signal: abortSignal,
    });

    if (!response.ok) {
      throw new Error(`API error: ${response.status}`);
    }

    if (!response.body) {
      throw new Error("Response body is empty");
    }

    let text = "";
    const toolCalls = new Map<string, any>();

    for await (const event of readSseEvents(response.body)) {
      if (event.type === "answer_delta") {
        text += event.message ?? "";
      }

      if (event.type === "tool_call_started") {
        toolCalls.set(event.data.tool_call_id, {
          type: "tool-call",
          toolCallId: event.data.tool_call_id,
          toolName: event.data.name,
          args: event.data.args ?? {},
          argsText: JSON.stringify(event.data.args ?? {}),
        });
      }

      if (event.type === "tool_call_finished") {
        const current = toolCalls.get(event.data.tool_call_id);

        if (current) {
          toolCalls.set(event.data.tool_call_id, {
            ...current,
            result: event.data.result_summary,
          });
        }
      }

      if (event.type === "error") {
        throw new Error(event.message ?? "Agent error");
      }

      yield {
        content: [
          ...Array.from(toolCalls.values()),
          ...(text ? [{ type: "text" as const, text }] : []),
        ],
      };
    }
  },
};
```

ここが今回の肝です。Python側では `answer_delta` を差分で送っていますが、assistant-ui の `LocalRuntime` へ `yield` するときは **累積済みの状態**を返します。公式ドキュメントでも、各 `yield` は前回の content を置き換えるので、delta ではなく full state を返す必要があると説明されています。([assistant-ui][1])

そのため、この部分で `text += event.message` としています。

```tsx id="eskv4p"
if (event.type === "answer_delta") {
  text += event.message ?? "";
}
```

そして、毎回こう返します。

```tsx id="7ochz4"
yield {
  content: [
    ...Array.from(toolCalls.values()),
    ...(text ? [{ type: "text" as const, text }] : []),
  ],
};
```

これは「現在のassistant messageは、tool call parts と text part で構成されている」と assistant-ui に伝えています。tool call についても、公式ドキュメントでは streaming中に tool call を扱う場合、過去の tool call が消えないように `Map` などで蓄積する例が示されています。([assistant-ui][1])

## 3. FastAPI側との対応

FastAPI側は、前に話した `AgentEvent` を SSE で返します。

```python id="xvft5s"
import json
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()


def to_sse(event: AgentEvent) -> str:
    payload = {
        "type": event.type,
        "message": event.message,
        "data": event.data,
    }
    return f"data: {json.dumps(payload, ensure_ascii=False)}\n\n"


@app.post("/api/chat/stream")
async def chat_stream(req: ChatRequest):
    ctx = ExecContext()
    agent_req = Request.from_messages(req.messages)

    async def stream():
        async for event in agent.run_agent_stream(ctx, agent_req):
            yield to_sse(event)

    return StreamingResponse(
        stream(),
        media_type="text/event-stream",
    )
```

対応関係はこうです。

```text id="h2ibpe"
Python agent:
  yield AgentEvent(type="answer_delta", message="こんにちは")

FastAPI:
  data: {"type":"answer_delta","message":"こんにちは","data":{}}

React:
  text += event.message

assistant-ui:
  yield { content: [{ type: "text", text }] }

Thread:
  assistant messageとして表示
```

つまり、最終回答は別APIで取りに行きません。**同じ stream の中に流れてくる `answer_delta` を、React側で assistant-ui の text part に変換する**だけです。

## 4. Data Stream Protocol に寄せる場合

`Data Stream Protocol` に寄せる場合、React側はさらに薄くできます。assistant-ui の `@assistant-ui/react-data-stream` は Data Stream Protocol を消費する runtime で、これは `LocalRuntime` の上に重なっている標準的な message-streaming protocol です。公式ドキュメントでは、text、tool calls、conversation context、error handling、cancellation、attachments を扱えると説明されています。([assistant-ui][5])

React側はかなり短くなります。

```tsx id="q8gf1w"
// src/App.tsx

import { AssistantRuntimeProvider } from "@assistant-ui/react";
import { useDataStreamRuntime } from "@assistant-ui/react-data-stream";
import { Thread } from "@/components/assistant-ui/thread";

export default function App() {
  const runtime = useDataStreamRuntime({
    api: "/api/chat",
    protocol: "data-stream",
  });

  return (
    <AssistantRuntimeProvider runtime={runtime}>
      <div className="h-screen">
        <Thread />
      </div>
    </AssistantRuntimeProvider>
  );
}
```

assistant-ui の公式サンプルでも、`useDataStreamRuntime({ api: "/api/chat", protocol: "data-stream" })` を作り、それを `AssistantRuntimeProvider` に渡して `<Thread />` を表示する形が示されています。([assistant-ui][5])

ただし、この方式では FastAPI 側が `Data Stream Protocol` に合ったレスポンスを返す必要があります。公式ドキュメントでも、backend は POST を受け取り data stream response を返す必要があり、request body には `messages`、`tools`、`system`、`threadId` が含まれるとされています。([assistant-ui][5])

なので初心者向けの現実的な順番はこうです。

```text id="7r6l9f"
第一段階:
  LocalRuntime + 自作SSE
  → 仕組みを理解しやすい
  → Python側のAgentEventをそのまま活かせる

第二段階:
  Data Stream Protocol
  → React側は薄くなる
  → FastAPI側をassistant-ui標準stream形式に寄せる

第三段階:
  tool UI / approval / file attachments
  → assistant-uiの機能に乗せる
```

## 5. tool表示はどうするか

まずは `ToolFallback` のままでもよいです。assistant-ui の thread component には tool call 表示の fallback が組み込まれています。慣れてきたら、特定の tool だけ専用UIにします。

たとえば `search_web` の表示をカスタムしたい場合は、`makeAssistantToolUI` を使います。公式ドキュメントでは、`makeAssistantToolUI` は tool execution と status を描画する custom UI component を登録するための utility で、`args`、`result`、`status` を受け取ると説明されています。([assistant-ui][6])

```tsx id="0dsh48"
// src/tools/SearchWebToolUI.tsx

import { makeAssistantToolUI } from "@assistant-ui/react";

export const SearchWebToolUI = makeAssistantToolUI({
  toolName: "search_web",
  render: ({ args, result, status }) => {
    if (status.type === "running") {
      return (
        <div className="rounded-md border p-3 text-sm">
          検索しています: {JSON.stringify(args)}
        </div>
      );
    }

    if (status.type === "complete") {
      return (
        <div className="rounded-md border p-3 text-sm">
          検索が完了しました。
          <pre>{JSON.stringify(result, null, 2)}</pre>
        </div>
      );
    }

    if (status.type === "incomplete") {
      return (
        <div className="rounded-md border p-3 text-sm text-red-600">
          検索に失敗しました。
        </div>
      );
    }

    return null;
  },
});
```

これを provider の内側に置きます。

```tsx id="7b60vo"
import { Thread } from "@/components/assistant-ui/thread";
import { MyRuntimeProvider } from "./runtime/MyRuntimeProvider";
import { SearchWebToolUI } from "./tools/SearchWebToolUI";

export default function App() {
  return (
    <MyRuntimeProvider>
      <SearchWebToolUI />
      <div className="h-screen">
        <Thread />
      </div>
    </MyRuntimeProvider>
  );
}
```

`SearchWebToolUI` 自体は画面に直接何かを出すコンポーネントというより、assistant-ui に「`search_web` という tool call が来たらこのUIで描画して」と登録するものです。公式ドキュメントでも、この component 自体は何も render せず、Assistant に tool UI を登録するものだと説明されています。([assistant-ui][6])

## 6. React初心者が混乱しやすいところ

一番混乱しやすいのは、`yield` が2か所に出てくることです。

Python側。

```python id="dcfz4t"
async for event in agent.run_agent_stream(...):
    yield to_sse(event)
```

React / TypeScript側。

```tsx id="e5pvxo"
async *run(...) {
  for await (const event of readSseEvents(...)) {
    yield {
      content: [{ type: "text", text }],
    };
  }
}
```

これは同じ思想ですが、役割が違います。

```text id="iaayj7"
Python側のyield:
  AgentEventをHTTP streamへ流す

TypeScript側のyield:
  assistant-ui Runtimeへ現在のassistant message状態を渡す
```

また、assistant-ui の `LocalRuntime` では **deltaではなく現在の全文を返す** という点が重要です。これは通常のLLM streamingと感覚が違います。LLMは `delta` を返しますが、assistant-ui の `run` から `yield` する値は「現在のmessage全体」です。([assistant-ui][1])

## 8. 最初に目指す完成形

最初の完成形は、これで十分です。

```text id="rz3a59"
ユーザーがThreadの入力欄から送信
  ↓
assistant-ui が MyModelAdapter.run() を呼ぶ
  ↓
MyModelAdapter が FastAPI /api/chat/stream を fetch
  ↓
FastAPI が agent.run_agent_stream() を読む
  ↓
AgentEvent がSSEで流れる
  ↓
React側で AgentEvent を assistant-ui content に変換
  ↓
Thread に progress / tool / answer が表示される
```

この構成なら、React側は「UIを全部自作する」のではなく、**assistant-ui の Thread に対して、backend stream をどう渡すかだけを書く**ことになります。あなたの今の理解レベルなら、まず `LocalRuntime + MyModelAdapter` から入るのがかなり現実的です。その後、FastAPI側のレスポンス形式を assistant-ui の `Data Stream Protocol` に寄せれば、React側の custom parser を減らして `useDataStreamRuntime` に移行できます。
