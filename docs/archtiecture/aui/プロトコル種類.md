結論として、**Python + FastAPI の自作エージェントを assistant-ui のフロントエンドに接続するなら、最初は `LocalRuntime` が一番自然**です。assistant-ui 側がチャット状態を持ち、あなたは FastAPI に `messages` を投げて返すだけで始められます。assistant-ui の公式説明でも、`LocalRuntime` は「custom backend に接続する最も簡単な方法」で、`ChatModelAdapter` の `run` を1つ実装すればよい位置づけです。([assistant-ui][1])

assistant-ui における **Runtime** は、UI コンポーネントとバックエンド・LLM をつなぐ層です。UI は Runtime 経由で message、composer、thread、tool call、再生成、編集などを扱います。つまり Runtime は「React UI と Agent API の変換・状態管理レイヤー」と考えると分かりやすいです。([assistant-ui][2])

| Runtime                                               | 何向けか                                              | Python + FastAPI での見方                                |
| ----------------------------------------------------- | ------------------------------------------------- | ---------------------------------------------------- |
| `LocalRuntime`                                        | 独自 REST API / 独自 LLM API に接続                      | **最初に選ぶ候補**。FastAPI `/api/chat` に `fetch` するだけで始めやすい |
| `DataStreamRuntime`                                   | バックエンドが data stream / UI message stream を返す場合     | FastAPI で streaming をちゃんと実装したい段階で検討                  |
| `AssistantTransport`                                  | メッセージだけでなく、agent の状態全体を UI に流したい場合                | 自作 agent framework に近いものを作るなら有力。ただし設計は重め             |
| `ExternalStoreRuntime`                                | Redux / Zustand / TanStack Query などでフロント側が状態を持つ場合 | 最初は不要。フロントで状態管理を完全に握りたい場合向け                          |
| `useChatRuntime` / AI SDK Runtime                     | Vercel AI SDK / Next.js API Route 前提に近い           | Python + FastAPI だけなら第一候補ではない。Node/AI SDK に寄せるならあり   |
| `AG-UI Runtime`                                       | AG-UI protocol の agent server に接続                 | Python agent を AG-UI protocol に合わせるなら選択肢             |
| LangGraph / LangChain / Google ADK / A2A / OpenCode 系 | その framework / protocol を使っている場合                  | 使っていないなら選ばない                                         |

あなたの構成だと、まずこう考えるのがよいです。

```text
React + assistant-ui
  ↓
LocalRuntime
  ↓ fetch
FastAPI /api/chat
  ↓
自作 Python Agent
  ↓
LiteLLM / OpenAI互換API / ToolCall / File処理
```

最小構成はこの形です。

```tsx
"use client";

import { AssistantRuntimeProvider, useLocalRuntime } from "@assistant-ui/react";
import type { ChatModelAdapter } from "@assistant-ui/react";
import { Thread } from "@/components/assistant-ui/thread";

const modelAdapter: ChatModelAdapter = {
  async run({ messages, abortSignal }) {
    const res = await fetch("http://localhost:8000/api/chat", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ messages }),
      signal: abortSignal,
    });

    const data = await res.json();

    return {
      content: [{ type: "text", text: data.text }],
    };
  },
};

export function ChatPage() {
  const runtime = useLocalRuntime(modelAdapter);

  return (
    <AssistantRuntimeProvider runtime={runtime}>
      <Thread />
    </AssistantRuntimeProvider>
  );
}
```

FastAPI 側はまずこれで十分です。

```py
from fastapi import FastAPI
from pydantic import BaseModel

    messages: list[dict[str, Any]]

@app.post("/api/chat")
async def chat(req: ChatRequest):

    result_text = "こんにちは。FastAPI からの応答です。"
    return {"text": result_text}
```

`DataStreamRuntime` は、最初からストリーミングを assistant-ui の protocol に合わせたい場合の選択肢です。公式では `@assistant-ui/react-data-stream` が data stream protocol を消費し、streaming text、tool calls、error handling、cancellation、attachments などに対応すると説明されています。ただし、バックエンド側が protocol に合う形式で返す必要があります。([assistant-ui][3])

`AssistantTransport` は、あなたが作っているような「自作 AI エージェント」にかなり近い考え方です。メッセージの断片ではなく、agent の状態スナップショットを stream し、フロントからは `add-message` や `add-tool-result` のような command を送る設計です。公式でも custom agent framework や、内部状態を UI に出したい場合に使うものとして説明されています。([assistant-ui][4])

ただし、今の段階では **いきなり `AssistantTransport` に行かない方がよい** と思います。理由は、あなたの現状のコアはすでに `messages → agent実行 → response` の形で動いているはずなので、まず `LocalRuntime` で接続した方が、UI 統合の確認が速いからです。その後、tool call の途中状態、approval、人間確認、実行ログ、状態遷移を UI に強く出したくなった段階で `AssistantTransport` または `AG-UI Runtime` を検討するのが自然です。

`AI SDK Runtime` は公式では新規なら v6 が現行で、`useChatRuntime` が推奨されています。これは Vercel AI SDK の `useChat` を包み、Cloud thread support や adapter、`AssistantChatTransport` を追加するものです。([assistant-ui][5]) ただし Python + FastAPI を主軸にするなら、AI SDK の流儀に合わせる必要が出るため、最初の選択肢としてはやや遠回りです。

整理すると、あなた向けの判断はこれです。

```text
まず動かす:
  LocalRuntime

FastAPI で streaming もやりたい:
  LocalRuntime の async generator
  または DataStreamRuntime

agent の内部状態・tool状態・承認・HITLをUIに深く出したい:
  AssistantTransport
  または AG-UI Runtime

Next.js + Vercel AI SDK に寄せる:
  useChatRuntime

Redux/Zustand でフロント状態を握りたい:
  ExternalStoreRuntime
```

なので最初の実装方針は、**assistant-ui + `LocalRuntime` + FastAPI `/api/chat`** でよいです。ここで UI 接続を固めてから、streaming、tool call 表示、approval UI、thread 保存を足していくのが一番見通しがよいです。

結論として、**LocalRuntime で streaming できます。HITL が必要なら、むしろ LocalRuntime を選ぶのが自然**です。

assistant-ui 公式でも、`LocalRuntime` は `ChatModelAdapter` の `run` を `async *run` にすると streaming できる、と説明されています。各 `yield` では差分ではなく「その時点までの累積 content」を返す必要があります。([assistant-ui][1])

一方で、`DataStreamRuntime` は streaming 用に見えますが、公式ドキュメント上では **Human-in-the-loop tools は DataStreamRuntime では未対応で、必要なら LocalRuntime を直接使う** と書かれています。([assistant-ui][2]) なので、あなたの要件が「FastAPI + streaming + HITL」なら、まずはこれです。

```text
React + assistant-ui
  ↓
LocalRuntime
  ↓ async *run で fetch stream を読む
FastAPI StreamingResponse
  ↓
自作 Python Agent
```

つまり、`DataStreamRuntime` は「assistant-ui が決めた streaming protocol にバックエンドを合わせる」方式です。`LocalRuntime` は「自分で fetch して、自分で assistant-ui の message content に変換する」方式です。HITL が必要なら後者の方が制御しやすいです。

最小イメージはこうです。

```tsx
const MyModelAdapter: ChatModelAdapter = {
  async *run({ messages, abortSignal }) {
    const res = await fetch("http://localhost:8000/api/chat/stream", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ messages }),
      signal: abortSignal,
    });

    if (!res.body) {
      throw new Error("No response body");
    }

    const reader = res.body.getReader();
    const decoder = new TextDecoder();

    let text = "";

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;

      const chunk = decoder.decode(value, { stream: true });
      text += chunk;

      // LocalRuntime では差分ではなく累積テキストを返す
      yield {
        content: [{ type: "text", text }],
      };
    }
  },
};
```

FastAPI 側はまず単純な streaming でよいです。

```py
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from typing import Any, AsyncGenerator

app = FastAPI()

class ChatRequest(BaseModel):
    messages: list[dict[str, Any]]

async def agent_stream(req: ChatRequest) -> AsyncGenerator[str, None]:
    # 本来はここで自作 Agent / LiteLLM stream を読む
    yield "こんにちは。"
    yield "これは "
    yield "streaming response です。"

@app.post("/api/chat/stream")
async def chat_stream(req: ChatRequest):
    return StreamingResponse(
        agent_stream(req),
        media_type="text/plain",
    )
```

HITL は LocalRuntime 側で扱えます。公式では `unstable_humanToolNames` に指定した tool はコードで即実行されず、tool call のところで run を止め、ユーザーが UI から結果を返す形になります。run を止めるには adapter 側が `status: { type: "requires-action", reason: "tool-calls" }` を返す必要があります。([assistant-ui][1])

概念的にはこうです。

```text
1. Agent が send_email tool call を出す
2. assistant-ui に tool-call part を表示する
3. status: requires-action で止める
4. ユーザーが Allow / Deny を押す
5. LocalRuntime が adapter を再実行する
6. adapter が承認結果を読んで FastAPI に resume / execute を投げる
7. backend が実行して続きの assistant message を返す
```

メール送信、ファイル削除、DB更新、外部API実行、見積作成確定のような操作があるなら、この HITL は入れた方がよいです。特に製造業向けエージェントなら、「AI が勝手に実行した」よりも「AI が提案し、人が承認し、実行ログが残る」構成の方が現実的です。

なので判断はこうです。

```text
streaming だけ必要
→ LocalRuntime でも DataStreamRuntime でも可能

streaming + HITL が必要
→ LocalRuntime が第一候補

backend が assistant-ui / AI SDK 系の data stream protocol に完全対応している
→ DataStreamRuntime も候補

agent state 全体、承認待ち、実行ログ、状態遷移を protocol として整理したい
→ 将来的に AssistantTransport / AG-UI を検討
```

今のあなたのケースでは、**LocalRuntime で進めてよい**です。`DataStreamRuntime` に寄せると streaming はきれいになりますが、HITL で詰まりやすいので、最初から選ばない方がよいと思います。
